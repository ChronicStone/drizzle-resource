`drizzle-resource` is a query layer for [Drizzle ORM](https://orm.drizzle.team) that standardizes how your server handles list-style endpoints.

You define a **resource** once. Every endpoint backed by that resource gets filters, search, sorting, pagination, row hydration, and facets — all driven by the same typed request contract.

## The problem without it

Building a table endpoint from scratch usually looks something like this:

```ts twoslash
// @noErrors
import { and, count, db, eq, ilike, or, orders } from "./twoslash-support";

const req = {
  query: {
    dir: "asc",
    page: "1",
    pageSize: "20",
    search: "acme",
    sort: "createdAt",
  },
};

const ctx = {
  user: {
    orgId: "acme",
  },
};

const countRows = [{ count: 42 }];
const orderBy = [];
// ---cut---
// somewhere in a controller...
const page = Number(req.query.page) || 1
const pageSize = Number(req.query.pageSize) || 20
const sort = req.query.sort || 'createdAt'
const dir = req.query.dir === 'asc' ? 'asc' : 'desc'
const search = req.query.search as string

const where = and(
  eq(orders.orgId, ctx.user.orgId),
  search ? or(ilike(orders.reference, `%${search}%`), ...) : undefined,
)

const [rows, [{ count }]] = await Promise.all([
  db.query.orders.findMany({
    with: { customer: true },
  limit: pageSize,
    // @ts-expect-error example placeholder
    offset: (page - 1) * pageSize,
    // @ts-expect-error example placeholder
    orderBy,
  }),
  Promise.resolve(countRows),
])
```

This works for one endpoint. But you write a version of it for every table. The sort params differ. The search fields differ. The filter logic differs. The count query is sometimes wrong. And when the client wants filter buckets for a sidebar, you add another parallel endpoint.

**That is the problem `drizzle-resource` solves.**

## What it gives you instead

Three building blocks:

1. **`createQueryEngine({ db, schema, relations })`** — creates a typed engine bound to your Drizzle setup
2. **`engine.defineResource(root, options)`** — declares a resource with field rules, scope, search, sort, facets, and optional execution overrides
3. **`resource.query({ request, context })`** — runs a full page pipeline: normalize → scope → validate → ids → rows → facets

The request shape is stable and typed. The same contract works for a simple table, a faceted explorer, and an ids-first caching flow.

## What it does not do

- It does not replace Drizzle. It sits on top of it.
- It does not generate migrations, handle mutations, or manage auth.
- It is not an ORM adapter or a REST framework.

It is a query layer specifically for server-side list endpoints that need consistent, typed behavior across your codebase.

## When it is a good fit

Use it when you have multiple table endpoints and you want them to share a consistent request contract. Especially useful for:

- **Admin interfaces** with sortable columns, search bars, and filter panels
- **Faceted explorer UIs** where filter buckets update alongside the result list
- **APIs consumed by a frontend data-table component** that expects a stable request/response shape
- **Multi-tenant backends** where every query needs a scope constraint that cannot be bypassed

## When to look elsewhere

If you have a single endpoint with simple requirements and you want total control over the SQL, you do not need this. The value compounds when you have multiple resources that need to behave consistently.
