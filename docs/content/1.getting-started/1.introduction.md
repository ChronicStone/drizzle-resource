# Introduction

`@exassess/drizzle-query-resource` gives you one typed, schema-driven way to handle list-style server queries in front of Drizzle ORM.

It is designed for endpoints that need a stable public contract for:

- table and grid queries
- faceted search UIs
- ids-first, hydrate-later flows
- multi-tenant constraints
- custom SQL strategies without losing typed field paths

## Use one contract across your endpoints

The package revolves around three pieces:

1. `createQueryEngine({ db, schema, relations })`
2. `engine.defineResource(root, options)`
3. `resource.query*({ request, context })`

Each resource combines:

- a root table such as `'employees'`
- an optional Drizzle `relations.with` tree
- declarative query rules under `query`
- optional imperative overrides under `strategy`

## Keep your field paths typed

Field paths are inferred from your schema and declared relations, so you can expose nested paths like:

- `fullName`
- `department.name`
- `department.company.name`
- `employeeSkills.skill.label`

This keeps your query contract explicit while still supporting relation-aware search, filtering, sorting, and facets.

## Start with the built-in execution model

When you do not provide custom strategies, the package runs a full automatic pipeline:

- ids are resolved with the built-in SQL compiler
- row hydration uses Drizzle relational queries
- facets are resolved with grouped SQL per requested facet

This is the best starting point for most resources because it keeps the pipeline simple, typed, and consistent.

## Reach for overrides only when they earn their keep

The package includes escape hatches, but they are meant to be selective:

- use `strategy.ids` when ids and count are the hot path
- use `strategy.rows` when hydration shape or source needs to change
- use `strategy.facets` when grouped facet SQL becomes too expensive
- use `strategy.query` only when the staged pipeline no longer fits
