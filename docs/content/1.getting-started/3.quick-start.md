This guide walks you through the three core steps: create the engine, define a resource, run a query. We'll use an orders + products domain throughout.

## Step 1 — Create the engine

The engine is created once and shared across all your resources. It binds to your Drizzle database, schema, and relations.

```ts twoslash [engine.ts]
import { createQueryEngine } from "drizzle-resource";
import { db } from "./db";
import { schema } from "./schema";
import { relations } from "./relations";

export const engine = createQueryEngine({ db, schema, relations }).withContext<{ orgId: string }>();
```

`.withContext<T>()` declares the shape of your per-request context — in this case a tenant identifier. This context flows through scope functions and strategy overrides. You can omit it if you don't need per-request context.

## Step 2 — Define a resource

A resource declares the root table, which relations to join, and the field-level rules for search, sorting, filters, and facets.

```ts twoslash [orders.resource.ts]
import { engine } from "./engine";

export const ordersResource = engine.defineResource("orders", {
  // Which related tables to join when hydrating rows
  relations: {
    customer: true,
    orderLines: {
      with: { product: true },
    },
  },

  query: {
    // A scope filter that merges into every request automatically.
    // The client cannot override or remove it.
    scope: (f, ctx) => f.is("customer.orgId", ctx.orgId),

    // Which fields are valid for free-text search,
    // and which ones are searched when the client sends fields: []
    search: {
      allowed: ["reference", "customer.name", "orderLines.product.name"],
      defaults: ["reference", "customer.name"],
    },

    // Default sort when the request sends no sorting array
    sort: {
      defaults: [{ key: "createdAt", dir: "desc" }],
    },

    // Which fields can be used as facets
    facets: {
      allowed: ["status", "customer.name", "orderLines.product.category"],
    },

    // Default page size when the request sends no pagination
    defaults: {
      pagination: { pageSize: 25 },
    },
  },
});
```

## Step 3 — Run a query

Call `resource.query({ request, context })` from your controller. The engine normalizes, validates, executes the full pipeline, and returns typed rows + a row count.

```ts twoslash [orders.controller.ts]
import { ordersResource } from "./orders.resource";

const result = await ordersResource.query({
  // Per-request context — flows into the scope function
  context: { orgId: "acme" },

  request: {
    pagination: { pageIndex: 1, pageSize: 25 },
    sorting: [{ key: "createdAt", dir: "desc" }],

    // Free-text search. fields: [] means "use the resource's defaults"
    search: { value: "laptop", fields: [] },

    // Top-level filters are AND-ed together. Add groups when you need nesting.
    filters: [
      {
        type: "condition",
        key: "status",
        operator: "isAnyOf",
        value: ["pending", "processing"],
      },
    ],

    context: {},

    // Optional: request facet buckets alongside the page
    facets: [
      { key: "status", mode: "exclude-self", limit: 10 },
      { key: "customer.name", mode: "include-self", limit: 20 },
    ],
  },
});

// result.rows     — typed array of hydrated order rows
// result.rowCount — total matching rows (before pagination)
// result.facets   — bucket arrays for each requested facet
```

## The four query methods

Every resource exposes four entry points:

| Method                      | Use when                                                   |
| --------------------------- | ---------------------------------------------------------- |
| `resource.query(...)`       | Full pipeline — ids + rows + optional facets in one call   |
| `resource.queryIds(...)`    | You only need page ids and a count (e.g. to cache them)    |
| `resource.queryRows(...)`   | You already have ordered ids and just need to hydrate them |
| `resource.queryFacets(...)` | You want facets independently from the page                |

Start with `query`. Move to the narrower methods only when your caching or batching flow needs them.
