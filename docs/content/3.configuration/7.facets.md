# Facets

This page is about configuration, not the facet request payload itself.

Facet behavior lives under `query.facets`.

## Restrict facet requests with `allowed`

```ts [resource.ts]
query: {
  facets: {
    allowed: [
      'department.name',
      'department.company.name',
      'employeeSkills.skill.label',
    ],
  },
}
```

If omitted, every registered field is considered facetable.

## Understand the built-in facet behavior

The built-in resolver supports:

- `exclude-self` mode by default
- optional bucket search
- optional bucket pagination via `limit` and `nextCursor`
- relation and many-to-many field paths

## Configure facet policy and custom execution together

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
    employeeSkills: {
      with: {
        skill: true,
      },
    },
  },
  query: {
    facets: {
      allowed: [
        'department.name',
        'department.company.name',
        'employeeSkills.skill.label',
      ],
    },
  },
  strategy: {
    facets: async ({ facets }) => ({
      facets: facets.map((facet) => ({
        key: facet.key,
        options: [],
        nextCursor: null,
      })),
    }),
  },
})
```

## Treat relation-heavy facets carefully

General rules of thumb:

- root-column facets are usually cheapest
- single-valued relation facets are usually manageable
- many-to-many facets are usually the most expensive

## Validate facet fields like the rest of the contract

Unknown facet fields are rejected before execution.
