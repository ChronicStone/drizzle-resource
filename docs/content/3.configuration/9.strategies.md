# Strategies

Strategies are the escape hatches of the package.

Use them when the request contract is already correct, but the built-in execution plan is not the right fit for your endpoint, SQL shape, or row shape.

## Understand the default engine pipeline first

Before overriding anything, it helps to know what the engine does by default:

1. It merges `query.scope` into the incoming request filters.
2. It normalizes pagination and fallback search or sorting defaults.
3. It validates requested fields against the resource field registry.
4. It resolves ordered page ids and total count.
5. It hydrates rows from those ids.
6. It resolves facets if the request asked for them.

That means strategy hooks do not replace configuration. They replace one or more execution stages after the request has already been normalized and validated.

## Choose the smallest override that solves the problem

- Use `strategy.ids` when page selection and count are the bottleneck.
- Use `strategy.rows` when hydration shape or source must change.
- Use `strategy.facets` when the built-in grouped facet queries are too expensive or semantically wrong.
- Use `strategy.query` only when the whole staged pipeline no longer maps well to the endpoint.

## Know every strategy function you can provide

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
    employeeSkills: {
      with: {
        skill: true,
      },
    },
  },
  strategy: {
    query: async (args) => ({ rows: [], rowCount: 0, facets: [] }),
    ids: async (args) => ({ ids: [], rowCount: 0 }),
    rows: async (args) => [],
    facets: async (args) => ({ facets: [] }),
  },
})
```

Only implement the hook you actually need.

## Override `ids` when page selection is the hot stage

`strategy.ids(args)` is usually the first override worth trying.

It is the best fit when:

- ids and count dominate latency
- relation-path sorting or filtering causes expensive joins
- you still want built-in row hydration

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
    employeeSkills: {
      with: {
        skill: true,
      },
    },
  },
  strategy: {
    ids: async ({ request, utils }) => {
      const whereClause = utils.buildWhereClause(request)
      const orderBy = utils.compileOrderBy(request.sorting)

      // Use your own SQL here, but keep the same public return contract.
      void whereClause
      void orderBy

      return {
        ids: ['emp_1', 'emp_2'],
        rowCount: 2,
      }
    },
  },
})
```

Why this hook matters:

- it preserves the resource contract
- it preserves default rows hydration unless you also override `rows`
- it lets you replace the most expensive stage without taking over the whole pipeline

## Override `rows` when the built-in hydration shape is wrong

`strategy.rows(args)` is for changing how rows are assembled after ids have already been chosen.

Use it when:

- you want a projection instead of Drizzle relational rows
- you need denormalized rows
- you need to hydrate from another service or backend

```ts [resource.ts]
const employees = engine.defineResource<
  'employees',
  {
    department: {
      with: {
        company: true,
      },
    },
  },
  { orgId: string },
  { id: string; fullName: string; skillCount: number }
>('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
  },
  strategy: {
    rows: async ({ ids }) =>
      ids.map((id) => ({
        id,
        fullName: `Employee ${id}`,
        skillCount: 4,
      })),
  },
})
```

This hook only receives ids that were already selected and ordered.

## Override `facets` when facet resolution is the problem

`strategy.facets(args)` replaces the facet stage used by both:

- `resource.queryFacets(...)`
- `resource.query(...)` when `request.facets` is present

Use it when:

- built-in per-facet grouped SQL is too expensive
- you need precomputed buckets
- you need custom aggregation semantics

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
    employeeSkills: {
      with: {
        skill: true,
      },
    },
  },
  strategy: {
    facets: async ({ facets }) => ({
      facets: facets.map((facet) => ({
        key: facet.key,
        options: [],
        nextCursor: null,
      })),
    }),
  },
})
```

## Override `query` only when one custom backend flow should own everything

`strategy.query(args)` skips the built-in `ids -> rows` pipeline entirely.

Use it when:

- one backend call should produce rows and total count together
- you need a totally custom execution backend
- the staged model of ids, then rows, then facets is no longer the right abstraction

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  strategy: {
    query: async () => ({
      rows: [{ id: 'emp_1', fullName: 'Ada Lovelace' }],
      rowCount: 1,
      facets: [],
    }),
  },
})
```

If `strategy.query` already returns `facets`, the package does not call `strategy.facets` again.

## Understand the exact args each hook receives

### `strategy.query(args)` and `strategy.ids(args)`

```ts [types.ts]
{
  request: QueryRequest
  context?: TContext
  resource: QueryResource<...>
  utils: QueryResourceUtils<TRow>
}
```

### `strategy.rows(args)`

```ts [types.ts]
{
  request: QueryRequest
  ids: Array<TRow extends { id: infer TId } ? TId : unknown>
  context?: TContext
  resource: QueryResource<...>
  utils: QueryResourceUtils<TRow>
}
```

### `strategy.facets(args)`

```ts [types.ts]
{
  request: QueryRequest
  facets: QueryFacetRequest<TFacetKey>[]
  context?: TContext
  resource: QueryResource<...>
  utils: QueryResourceUtils<TRow>
}
```

The important detail is that `request` is already normalized and validated by the engine before your strategy runs.

## Understand the `utils` helpers instead of treating them like magic

The utility object exists so custom strategies can stay aligned with the standard request contract.

### `utils.normalizeString(value)`

Use this when you need the same lowercase normalization behavior the built-in engine uses for case-insensitive matching.

### `utils.compileCondition(condition)`

Use this when you want to turn one filter condition into a Drizzle SQL fragment.

### `utils.compileFilterNode(node)`

Use this when you want the SQL for a full filter tree instead of only one condition.

### `utils.compileSearch(search)`

Use this when you want the exact built-in SQL for the search portion of the request.

### `utils.compileOrderBy(sorting)`

Use this when your custom ids query still wants the standard sorting semantics.

### `utils.buildWhereClause(request)`

Use this when you want one helper that combines:

- compiled search
- compiled filters

into the same `where` clause the default engine would use.

### `utils.executeIdsQuery({ request })`

Use this when you want to wrap the default ids behavior instead of fully replacing it.

### `utils.executeRowsQuery({ ids, request })`

Use this when you want built-in rows hydration after doing custom ids logic yourself.

### `utils.executeHydratedPage({ request })`

Use this when your custom hook wants the full default page result and then adds extra behavior around it.

### `utils.resolveFacets({ request, facets })`

Use this when you want the built-in facet behavior but need to pre-process or post-process around it.

### `utils.resolveField(path)`

Use this when your strategy needs metadata about one field from the resolved field registry.

## Use the helpers thoughtfully

This is a common pattern for a focused `ids` override:

```ts [resource.ts]
strategy: {
  ids: async ({ request, utils }) => {
    const whereClause = utils.buildWhereClause(request)
    const orderBy = utils.compileOrderBy(request.sorting)

    void whereClause
    void orderBy

    return {
      ids: ['emp_3', 'emp_8'],
      rowCount: 2,
    }
  },
}
```

This is a common pattern when you want custom ids selection but default hydration:

```ts [resource.ts]
strategy: {
  rows: async ({ ids, request, utils }) => {
    return utils.executeRowsQuery({ ids, request })
  },
}
```

## Keep your override as narrow as possible

The best strategy customizations feel boring:

- they preserve the public API
- they reuse built-in helpers where possible
- they document why they exist
- they replace only the stage that actually needed replacing
