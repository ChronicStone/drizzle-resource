# Search

This page is about configuration, not the search request payload itself.

Search behavior lives under `query.search`.

## Restrict searchable fields with `allowed`

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
  },
  query: {
    search: {
      allowed: [
        'fullName',
        'email',
        'department.name',
        'department.company.name',
      ],
    },
  },
})
```

If omitted, every registered field is considered searchable.

## Set fallback fields with `defaults`

```ts [resource.ts]
query: {
  search: {
    allowed: [
      'fullName',
      'email',
      'department.name',
      'department.company.name',
    ],
    defaults: ['fullName', 'department.company.name'],
  },
}
```

When `request.search.fields` is an empty array, the package falls back to `query.search.defaults`.

## Configure allowed and default search fields together

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
  },
  query: {
    search: {
      allowed: [
        'fullName',
        'email',
        'department.name',
        'department.company.name',
      ],
      defaults: ['fullName', 'department.company.name'],
    },
  },
})
```

## Keep search fields deliberate

Prefer fields that map cleanly to user expectations and stable SQL plans.

Be careful with:

- many-to-many paths in hot endpoints
- deeply nested relation paths
- fields that look searchable but should stay internal
