# Scope

`query.scope` adds filters automatically to every request.

## Use scope for constraints callers should not control

Typical examples:

- tenant isolation
- authorization rules
- soft-delete filtering
- visibility constraints

```ts [scope.ts]
const employeeScope = engine.defineScope(
  'employees',
  {
    department: {
      with: {
        company: true,
      },
    },
  },
  (ctx, filters) => filters.and([filters.is('department.company.country', ctx.orgId)]),
)
```

## Combine multiple automatic constraints in one scope

```ts [scope.ts]
const employeeScope = engine.defineScope(
  'employees',
  {
    department: {
      with: {
        company: true,
      },
    },
  },
  (ctx, filters) =>
    filters.and([
      filters.is('department.company.country', ctx.orgId),
      filters.isNot('email', ''),
    ]),
)
```

## Reuse typed scope builders across resources

```ts [resource.ts]
const employees = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
  },
  query: {
    scope: employeeScope,
  },
})
```

## See what scope changes at query time

```ts [query.ts]
await employees.query({
  context: {
    orgId: 'fr',
  },
  request: {
    pagination: { pageIndex: 1, pageSize: 25 },
    sorting: [{ key: 'fullName', dir: 'asc' }],
    search: { value: 'ada', fields: [] },
    filters: {
      type: 'group',
      combinator: 'and',
      children: [
        {
          type: 'condition',
          key: 'department.name',
          operator: 'is',
          value: 'Engineering',
        },
      ],
    },
    context: {},
  },
})
```

The caller only sends the department filter, but the scope still injects the tenant constraint automatically.

## Keep scope logic predictable

We recommend that scope handlers:

- stay focused on cross-cutting constraints
- avoid endpoint-specific branching when possible
- preserve the same public field-path rules as the rest of the resource
