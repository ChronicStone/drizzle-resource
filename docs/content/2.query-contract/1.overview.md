# Overview

The query contract is the client-facing request language of a resource.

It answers a different question from configuration:

- Query contract: what can the client send?
- Configuration: what does the resource allow, default, hide, or override?

## Start with how the engine sees a request

When a caller hits `resource.query(...)`, the engine does not execute the raw payload immediately.

Instead it goes through a consistent flow:

1. It reads the incoming `request`.
2. It merges automatic scope filters from `query.scope`.
3. It normalizes fallback pagination, sorting, and default search fields.
4. It validates requested fields against the resource field registry.
5. It runs ids selection, row hydration, and optional facet resolution.

That means the query contract is not just a payload shape. It is the normalized input the rest of the engine is built around.

## See the full request shape first

```ts [query-request.ts]
type QueryRequest = {
  pagination: {
    pageIndex: number
    pageSize: number
  }
  sorting: Array<{
    key: string
    dir: 'asc' | 'desc'
  }>
  filters: QueryFilterGroup
  search: {
    value: string
    fields: string[]
  }
  context: Record<string, unknown>
  facets?: QueryFacetRequest[]
}
```

## Read the contract like a UI author

If you are building a table, search screen, or faceted explorer, this section is the one you want.

It explains:

- what a filter tree looks like
- what search payloads look like
- what sorting rules the client sends
- how facet requests are structured
- what pagination means on the wire

It is not about how to configure those rules. That lives in [Configuration](/configuration/overview).

## Start from a realistic request

```ts [query.ts]
await employees.query({
  context: {
    orgId: 'fr',
  },
  request: {
    pagination: {
      pageIndex: 1,
      pageSize: 25,
    },
    sorting: [{ key: 'fullName', dir: 'asc' }],
    search: {
      value: 'ada',
      fields: [],
    },
    filters: {
      type: 'group',
      combinator: 'and',
      children: [
        {
          type: 'condition',
          key: 'department.name',
          operator: 'is',
          value: 'Engineering',
        },
      ],
    },
    context: {},
    facets: [
      {
        key: 'department.name',
        mode: 'exclude-self',
        limit: 10,
      },
    ],
  },
})
```

This one request drives all of these behaviors together:

- page selection
- field ordering
- search matching
- filter narrowing
- optional facet resolution
