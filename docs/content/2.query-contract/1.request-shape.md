# Request Shape

Every call to `resource.query(...)` accepts two top-level keys:

```ts twoslash
// ---cut-start---
const T = {} as const;
const QueryRequest = {} as import("drizzle-resource").QueryRequest;
declare function resourceQuery(input: {
  context: unknown;
  request: import("drizzle-resource").QueryRequest;
}): void;
const resource = { query: resourceQuery };
// ---cut-end---
resource.query({
  context: T, // your per-request context (tenant id, user, etc.)
  request: QueryRequest,
});
```

The `context` value flows into your scope function and strategy overrides. The `request` describes what the client wants: which page, in what order, with which filters, searching what.

## Full type

```ts twoslash
import type { QueryFacetRequest, QueryFilterNode } from "drizzle-resource";

type QueryRequest = {
  pagination: {
    pageIndex: number; // 1-based. Page 1 is the first page.
    pageSize: number;
  };
  sorting: Array<{
    key: string; // a valid field path on this resource
    dir: "asc" | "desc";
  }>;
  filters: QueryFilterNode[]; // top-level filters are AND-ed together
  search: {
    value: string; // the search string
    fields: string[]; // which fields to search; [] means use resource defaults
  };
  context: Record<string, unknown>; // extra client-side context (rarely used)
  facets?: QueryFacetRequest[]; // optional: request bucket counts
};
```

## Field-by-field

### `pagination`

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "pagination"> = {
  // ---cut---
  pagination: { pageIndex: 1, pageSize: 25 },
};
```

`pageIndex` is **1-based**. Sending `pageIndex: 1` returns the first page. The engine converts this to a SQL `OFFSET` of `(pageIndex - 1) * pageSize`.

If you omit `pagination` or send `pageSize: 0`, the resource's `defaults.pagination.pageSize` applies (typically 25 if you set it, otherwise the engine's fallback).

### `sorting`

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "sorting"> = {
  // ---cut---
  sorting: [{ key: "createdAt", dir: "desc" }],
};
```

Multiple sort entries are applied in order. The `key` must be a field path that exists in the resource's field registry — paths that resolve through declared relations are valid (`customer.name`, `orderLines.product.category`).

::warning
Sorting on many-to-many relation paths forces a join that can be expensive at scale. See [Performance → Field-type costs](/performance/optimization#field-type-costs) before exposing m2m sort keys to clients.
::

If `sorting` is empty, the resource's `sort.defaults` apply. If the resource has no sort defaults, the engine applies no ORDER BY (result order is database-defined).

If a sort key is in the resource's `sort.disabled` list, it is silently dropped.

### `filters`

Top-level filters are expressed as an array. The engine combines them with an implicit `and`:

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "filters"> = {
  // ---cut---
  filters: [
    // conditions and nested groups go here
  ],
};
```

An empty array produces no WHERE clause beyond the scope filter. Use a `group` node only when you need nested `and` / `or` logic. See [Filters](/query-contract/filters) for the full syntax.

### `search`

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "search"> = {
  // ---cut---
  search: { value: "laptop", fields: [] },
};
```

`value` is the search string. It is matched case-insensitively using `ILIKE '%value%'` against each search field.

`fields` is which fields to search. An empty array means "use the resource's `search.defaults`". Sending specific field paths restricts search to those fields, as long as they appear in the resource's `search.allowed` list.

If `value` is empty, search is skipped entirely.

### `facets`

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "facets"> = {
  // ---cut---
  facets: [
    { key: "status", mode: "exclude-self", limit: 10 },
    { key: "customer.name", mode: "include-self", limit: 20 },
  ],
};
```

Optional. Each entry requests bucket counts for one field path. See [Facets](/query-contract/facets) for `mode` semantics and cost implications.

## What the engine does with a request

Before any SQL runs, the engine normalizes the request:

1. **Merges scope filters** — your `query.scope` function's output is prepended to the top-level filter array and AND-ed with the rest. The client cannot see or override it.
2. **Applies defaults** — if `sorting` is empty, `sort.defaults` is used. If `pagination.pageSize` is 0 or missing, `defaults.pagination.pageSize` is used. If `search.fields` is empty, `search.defaults` is used.
3. **Validates field paths** — every sort key, search field, filter key, and facet key is checked against the resource's field registry. Invalid paths are rejected with an error.

Only then does it run SQL.
