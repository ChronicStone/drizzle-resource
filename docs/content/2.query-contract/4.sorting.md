# Sorting

The client sends an array of sort entries. Each entry has a field path and a direction.

## The sort payload

```ts
sorting: [
  { key: "createdAt", dir: "desc" },
  { key: "reference", dir: "asc" },
];
```

Multiple entries are applied in order — in this example, rows are sorted by `createdAt` descending, with `reference` ascending as a tiebreaker.

The `key` must be a field path registered in the resource. Any field path that resolves through the resource's declared relations is valid — `customer.name`, `orderLines.product.category`, etc. — unless it is in the resource's `sort.disabled` list.

If `sorting` is empty or omitted, the resource's `sort.defaults` apply.

## How defaults work

```ts
sort: {
  defaults: [{ key: 'createdAt', dir: 'desc' }],
}
```

The defaults only apply when the client sends an empty `sorting` array. If the client sends any sort entry, defaults are not merged in — the client array is used as-is. This means there is no implicit "secondary sort" from defaults when the client specifies a sort.

## Disabled fields

Some fields should not be sortable — either because the SQL is expensive (many-to-many paths require a join) or because sorting on them would expose unwanted ordering:

```ts
sort: {
  disabled: ['orderLines.product.name'],
}
```

If the client sends a sort key that is in `disabled`, that entry is silently dropped. No error is returned. The remaining sort entries still apply.

## The cost of relation-path sorting

Sorting by a field on a related table requires a JOIN in the ids query. This is typically fine for direct relations (one-to-one, many-to-one):

```ts
// Fine: customer is a direct many-to-one relation
sorting: [{ key: "customer.name", dir: "asc" }];
```

But sorting through a many-to-many relation (a junction table) is a different story. Consider sorting orders by `orderLines.product.category`:

- The engine must join `order_lines` and `products`
- Multiple rows per order exist in `order_lines`
- The sort result is ambiguous without aggregation
- The join inflates intermediate result size

::warning
Avoid exposing many-to-many relation paths as sort keys unless you have benchmarked the cost and applied a custom `strategy.ids` that handles the join correctly. Add such paths to `sort.disabled` by default and only enable them if there is a clear need.
::

For direct relation sorting, the cost is generally acceptable. For deep or m2m paths, benchmark first.
