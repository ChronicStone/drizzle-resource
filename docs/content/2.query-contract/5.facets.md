# Facets

Facets are bucket counts — the "12 orders in pending, 4 in processing" sidebar counts that let users understand the distribution of their filtered results. They travel in the same request as the main query, resolved against the same filter universe.

## The facet request

```ts twoslash
// @noErrors
import type { QueryRequest } from "drizzle-resource";

const request: Pick<QueryRequest, "facets"> = {
  // ---cut---
  facets: [
    { key: "status", mode: "exclude-self", limit: 10 },
    { key: "customer.name", mode: "include-self", limit: 20 },
  ],
};
```

`facets` is optional. Omitting it entirely skips facet resolution and saves the associated queries.

Each facet entry specifies:

| Field   | Type                               | Description                                                         |
| ------- | ---------------------------------- | ------------------------------------------------------------------- |
| `key`   | `string`                           | A field path allowed by the resource's `facets.allowed` list        |
| `mode`  | `'exclude-self' \| 'include-self'` | Whether to apply the filter for this key when computing its buckets |
| `limit` | `number`                           | Max number of buckets to return                                     |

## Mode: `exclude-self` vs `include-self`

This controls how a facet's own active filter affects its bucket counts.

**`exclude-self`** — when the client has an active filter on this field, that filter is ignored when computing this facet's buckets. The counts reflect all other active filters but not this one.

This is the correct mode for "refinement" facets. If the user is filtering by `status: pending`, the status facet should still show all statuses and their counts — not just show `pending: 12` with everything else hidden. Without `exclude-self`, a user who selects one status would lose visibility of the other options.

```ts twoslash
// @noErrors
const note = `
// User has status: 'pending' active
// exclude-self -> status facet shows:
// pending: 12, processing: 4, shipped: 31, cancelled: 2

// include-self -> status facet shows:
// pending: 12  (everything else is 0 because filter is applied)
`;
```

**`include-self`** — all active filters including this one are applied when computing buckets. Useful for "drill-down" scenarios where you want to show remaining options after applying a filter.

## The facet response

```ts twoslash
// @noErrors
import { result } from "./twoslash-support";

result.facets = [
  {
    key: "status",
    options: [
      { value: "pending", count: 12 },
      { value: "processing", count: 4 },
      { value: "shipped", count: 31 },
    ],
    total: 3,
  },
];
```

`total` is the total number of distinct values for this field in the filtered result, regardless of `limit`. This lets you show "showing 10 of 47" in the UI.

## What the engine actually runs

For each requested facet, the engine runs a grouped aggregate query against the filtered result universe. The effective query is roughly:

```sql
SELECT field_value, count(*) as count
FROM root_table
[JOINs for relation path]
WHERE [scope + client filters, minus self if exclude-self]
GROUP BY field_value
ORDER BY count DESC
LIMIT :limit
```

Plus a count-of-distinct-values query to populate `total`.

## Facet cost by field type

Not all facet paths cost the same. The cost depends on the relation type:

**Root-column facets** (e.g. `status`, `createdAt`) — cheapest. No join needed. The GROUP BY runs directly on the root table.

**Direct relation facets** (e.g. `customer.name` via a many-to-one join) — moderate. One JOIN, then GROUP BY on the joined column. Usually fast when indexed.

**Many-to-many facets** (e.g. `orderLines.product.category` through a junction table) — most expensive. Requires joining through the junction table, with `count(distinct root_id)` to avoid inflated counts. Under concurrency, these can dominate your query time.

::warning
Many-to-many facets are the most common source of performance problems with faceted queries. Benchmark them explicitly before exposing them to production traffic. See [Performance → Benchmark results](/performance/benchmarks) for real numbers.
::

## Restricting which facets clients can request

The resource's `facets.allowed` list controls which field paths are valid facet keys:

```ts twoslash
// @noErrors
import type { ExampleFacetsConfig } from "./twoslash-support";

const queryConfig = {
  // ---cut---
  facets: {
    allowed: ["status", "customer.name", "orderLines.product.category"],
  },
} satisfies { facets: ExampleFacetsConfig };
```

A client-requested facet key not in `allowed` is rejected. This prevents clients from requesting expensive or sensitive aggregations you haven't explicitly exposed.

## Requesting facets independently

If your UI updates the facets sidebar separately from the main page (e.g. on an initial load without a search), use `resource.queryFacets(...)` to resolve facets without running the full ids + rows pipeline:

```ts twoslash
// @noErrors
import { ordersResource, request } from "./twoslash-support";

const facets = await ordersResource.queryFacets({
  context: { orgId: "acme" },
  request,
  facets: [{ key: "status", mode: "exclude-self", limit: 10 }],
});
```
