# Pagination

## The pagination payload

```ts
pagination: {
  pageIndex: number; // 1-based. Page 1 is the first page.
  pageSize: number; // number of rows per page
}
```

`pageIndex` is **1-based**. Sending `pageIndex: 1` returns the first page, not `pageIndex: 0`. The engine translates this to a SQL `OFFSET` of `(pageIndex - 1) * pageSize`.

## Defaults

If your resource defines a `defaults.pagination.pageSize`, it applies when the client sends `pageSize: 0` or omits it entirely:

```ts
// In your resource definition
defaults: {
  pagination: { pageSize: 25 },
}
```

There is no built-in default for `pageIndex` — if omitted, page 1 is assumed.

## What the engine returns

The response always includes the current page's rows and the total count of matching rows:

```ts
{
  rows: TRow[],     // the current page
  rowCount: number, // total rows matching the filters (before pagination)
}
```

Use `rowCount` to compute total pages: `Math.ceil(rowCount / pageSize)`.

## The cost of deep pagination

Offset-based pagination gets more expensive as `pageIndex` increases. A query for page 500 with `pageSize: 25` requires the database to skip 12,475 rows before returning the next 25. The database cannot jump directly to row 12,476 — it must process all rows before it.

For a resource with a custom `strategy.ids`, you can write a more efficient ids query using keyset pagination (WHERE id > lastSeenId) instead of OFFSET. The rows pipeline stays the same.

::tip
If your users routinely access deep pages (analytics exports, bulk operations), consider splitting the flow: use `queryIds` with a custom strategy for efficient id selection, then `queryRows` for hydration. For typical user-driven table navigation (pages 1–10), offset pagination is fine.
::

## Page size limits

The engine does not enforce a maximum `pageSize` by default. Add an explicit cap in your controller or resource strategy if you want to prevent clients from requesting arbitrarily large pages:

```ts
// In your controller, before calling resource.query:
if (request.pagination.pageSize > 100) {
  request.pagination.pageSize = 100;
}
```

Or enforce it in a custom `strategy.ids` where you have full control over the SQL.
