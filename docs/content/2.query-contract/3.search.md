# Search

Search is a free-text input that matches against one or more field paths using case-insensitive substring matching.

## The search payload

```ts twoslash
// @noErrors
type SearchPayload = {
  // ---cut---
  search: {
    value: string; // the text to search for
    fields: string[]; // which fields to search; [] means use resource defaults
  };
};
```

When `value` is an empty string, search is skipped entirely and no ILIKE conditions are added to the query.

## How field selection works

The `fields` array lets the client control which fields to search against. There are three cases:

**Empty array** — the resource's `search.defaults` are used. This is the most common case. The frontend sends `fields: []` and the server decides what to search.

```ts twoslash
// @noErrors
import type { ExampleSearchConfig } from "./twoslash-support";

const queryConfig = {
  // ---cut---
  search: { value: "laptop", fields: [] },
} satisfies { search: ExampleSearchConfig };
```

**Specific fields** — the client sends explicit field paths. Each one is validated against the resource's `search.allowed` list. Fields not in `allowed` are rejected.

```ts twoslash
// @noErrors
import type { ExampleSearchConfig } from "./twoslash-support";

const queryConfig = {
  // ---cut---
  search: { value: "laptop", fields: ["orderLines.product.name"] },
} satisfies { search: ExampleSearchConfig };
```

**No search config on the resource** — if the resource defines no `search.allowed`, all registered field paths are considered searchable. This is rarely what you want in production; set `allowed` explicitly.

## How the SQL is compiled

Each search field becomes an `ILIKE '%value%'` condition. Multiple fields are joined with OR, meaning a row matches if the value appears in any of the searched fields.

For example, searching `'laptop'` across `reference` and `customer.name` produces:

```sql
WHERE (
  orders.reference ILIKE '%laptop%'
  OR customers.name ILIKE '%laptop%'
)
-- combined with scope and other filters using AND
```

When a search field resolves through a relation (like `orderLines.product.name`), the engine compiles the necessary JOIN. Many-to-many paths (through junction tables) can be expensive at scale — see [Performance → Field-type costs](/performance/optimization#field-type-costs).

## Defaults vs allowed

These are two separate concepts on the resource:

```ts twoslash
// @noErrors
import type { ExampleSearchConfig } from "./twoslash-support";

const queryConfig = {
  // ---cut---
  search: {
    allowed: ["reference", "customer.name", "orderLines.product.name", "orderLines.product.sku"],
    defaults: ["reference", "customer.name"],
  },
} satisfies { search: ExampleSearchConfig };
```

`defaults` should be the fields most users actually want to search — typically name-like columns on the root table and its direct relations. `allowed` is the full set the client can opt into.

::tip
Keep `defaults` narrow. Searching across many fields, especially through relations, adds cost to every request that uses free-text search. Start with root-table and direct-relation fields as defaults, and benchmark before adding many-to-many paths to defaults.
::

## Trimming and empty values

The engine trims whitespace from `value` before checking whether it is empty. A value of `'   '` behaves the same as `''` — no search is applied.
