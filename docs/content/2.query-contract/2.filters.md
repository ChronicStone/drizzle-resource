# Filters

Filters are expressed as a list of nodes. Top-level entries are combined with an implicit `and`, and you can use nested `group` nodes anywhere you need custom boolean logic.

## The two node types

### Condition

A single field comparison:

```ts
{
  type: 'condition',
  key: 'status',           // field path
  operator: 'is',          // see operator list below
  value: 'active',         // the comparison value
}
```

### Group

A logical combinator that wraps other nodes:

```ts
{
  type: 'group',
  combinator: 'and',       // 'and' | 'or'
  children: [
    // conditions or nested groups
  ],
}
```

## Full operator list

| Operator   | SQL equivalent    | Notes                            |
| ---------- | ----------------- | -------------------------------- |
| `is`       | `= value`         | Exact equality                   |
| `isNot`    | `!= value`        | Inequality                       |
| `isAnyOf`  | `IN (values)`     | `value` must be an array         |
| `contains` | `ILIKE '%value%'` | Case-insensitive substring       |
| `gt`       | `> value`         | Greater than                     |
| `gte`      | `>= value`        | Greater than or equal            |
| `lt`       | `< value`         | Less than                        |
| `lte`      | `<= value`        | Less than or equal               |
| `between`  | `BETWEEN a AND b` | `value` is `[a, b]`, inclusive   |
| `before`   | `< value`         | Alias for `lt`, useful for dates |
| `after`    | `> value`         | Alias for `gt`, useful for dates |

## Examples

### Simple AND filter

Orders in `pending` or `processing` status, created this year:

```ts
filters: [
  {
    type: 'condition',
    key: 'status',
    operator: 'isAnyOf',
    value: ['pending', 'processing'],
  },
  {
    type: 'condition',
    key: 'createdAt',
    operator: 'after',
    value: '2026-01-01T00:00:00.000Z',
  },
]
```

### Nested OR inside AND

Orders that are either high-value OR from a specific customer — and also not cancelled:

```ts
filters: [
  {
    type: 'condition',
    key: 'status',
    operator: 'isNot',
    value: 'cancelled',
  },
  {
    type: 'group',
    combinator: 'or',
    children: [
      {
        type: 'condition',
        key: 'totalAmount',
        operator: 'gte',
        value: 1000,
      },
      {
        type: 'condition',
        key: 'customer.name',
        operator: 'contains',
        value: 'acme',
      },
    ],
  },
]
```

### Filter on a nested relation field

Because field paths resolve through the resource's declared relations, you can filter on `customer.name`, `orderLines.product.category`, or any other path registered in the resource:

```ts
{
  type: 'condition',
  key: 'orderLines.product.category',
  operator: 'is',
  value: 'electronics',
}
```

The engine resolves the relation path and compiles the appropriate JOIN and WHERE clause.

## Empty filters

An empty array is valid and produces no WHERE clause beyond the scope:

```ts
filters: []
```

## Using the filter builder

When writing filters in server-side code (scope functions, strategy overrides), use the typed `createQueryFilterBuilder` DSL instead of constructing objects by hand:

```ts
import { createQueryFilterBuilder } from "drizzle-resource";

const f = createQueryFilterBuilder<MyFieldPaths>();

// Equivalent to the nested example above
f.and([
  f.isNot("status", "cancelled"),
  f.or([f.gte("totalAmount", 1000), f.contains("customer.name", "acme")]),
]);
```

The builder has methods for every operator: `f.is`, `f.isNot`, `f.isAnyOf`, `f.contains`, `f.gt`, `f.gte`, `f.lt`, `f.lte`, `f.between`, `f.before`, `f.after`, plus `f.and` and `f.or` for grouping.

See [Reference → Filter Builder](/reference/filter-builder) for the full API.

## Scope filters and client filters

Your resource's `query.scope` produces filters that are merged into the request's top-level `filters` array using AND. This happens before any SQL runs. The client-supplied filters are AND-ed alongside your scope — they cannot reach outside it.

```ts
// Effective WHERE is always:
// (scope condition) AND (client filters)
```

This is what makes scope the right place for tenancy and authorization constraints. See [Resource Setup → Scope](/resource-setup/scope).
