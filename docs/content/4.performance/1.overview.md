# Pipeline Overview

Understanding what SQL the engine runs helps you know where to look when performance is not where you want it.

## The five stages

```
request
  ↓
1. Normalize & validate   — apply defaults, merge scope, validate field paths
  ↓
2. Ids query              — SELECT id + COUNT(*) for the current page
  ↓
3. Rows query             — db.query.root.findMany({ with: relations })
  ↓
4. Facets (if requested)  — per-facet GROUP BY aggregations
  ↓
response: { rows, rowCount, facets? }
```

Only the ids and rows stages are mandatory. Facets are skipped entirely if `request.facets` is empty or absent.

## Stage 1 — Normalize & validate

No database work. The engine:

- merges `query.scope` filters into the request's filter group
- applies `sort.defaults` if `sorting` is empty
- applies `search.defaults` if `search.fields` is empty
- applies `defaults.pagination.pageSize` if not set
- validates every field path (sort keys, filter keys, search fields, facet keys) against the resource's registry

Validation errors surface here before any SQL runs.

## Stage 2 — Ids query

This is where the engine selects the page of matching row ids and the total count of matching rows.

The automatic ids query is a CTE-based SQL statement:

```sql
WITH matching_ids AS (
  SELECT orders.id
  FROM orders
  LEFT JOIN customers ON orders.customer_id = customers.id
  -- JOINs for filter/search/sort paths
  WHERE [scope + client filters + search ILIKE conditions]
  ORDER BY [sorting]
  LIMIT [pageSize] OFFSET [(pageIndex - 1) * pageSize]
)
SELECT
  matching_ids.id,
  (SELECT count(*) FROM matching_ids) AS total_count
FROM matching_ids
```

The CTE computes the filtered set once and reuses it for both page selection and count. This is cheaper than running two separate queries.

**This stage is the most impactful to optimize.** Relation-heavy ORDER BY clauses, many-to-many joins for search or filters, and large OFFSET values all accumulate cost here.

## Stage 3 — Rows query

Once the engine has the ordered id list from stage 2, it hydrates the full row objects:

```ts
db.query.orders.findMany({
  where: inArray(orders.id, ids),
  with: {
    customer: true,
    orderLines: { with: { product: true } },
  },
})
```

Drizzle fetches exactly the rows you asked for and joins the declared relations. With proper indexes on the primary key and foreign keys, this is fast regardless of how complex your join tree is — you are fetching a small, known set of rows.

The engine then restores the order from the ids array, since `inArray` does not guarantee order.

## Stage 4 — Facets

For each requested facet, the engine runs a grouped aggregate query:

```sql
-- Example: facet on status
SELECT
  orders.status AS value,
  count(*) AS count
FROM orders
LEFT JOIN customers ON orders.customer_id = customers.id
WHERE [same scope + client filters, minus self if exclude-self]
GROUP BY orders.status
ORDER BY count DESC
LIMIT [facet.limit]
```

Plus a count-of-distinct-values query for the `total` field in the response.

Facets share the filter universe with the main query but run independently from stages 2 and 3. The cost scales with:

- the number of requested facets (each is a separate query)
- the join complexity of each facet's field path
- the cardinality of the grouped field (high-cardinality fields like `customer.email` are expensive to aggregate)

## Where cost accumulates

| Scenario | Main cost driver | Where to look |
|---|---|---|
| Large table, no relations | OFFSET for deep pages, count query | `strategy.ids` with keyset pagination |
| Relation-heavy sort/filter | JOIN in ids query | `strategy.ids` with simpler join shape |
| Many-to-many search | Subquery for each ILIKE on m2m path | `search.defaults` — remove m2m paths |
| Multiple facets | One query per facet | `facets.allowed` — reduce cardinality, `strategy.facets` |
| m2m facet path | `count(distinct)` + junction join | `strategy.facets` or remove from `allowed` |

## The impact of good database indexes

Before adding custom strategies, ensure your database has indexes on:

- Foreign key columns used in JOIN conditions (`orders.customer_id`, `order_lines.order_id`)
- Columns used in WHERE conditions from common filters (`orders.status`, `orders.created_at`)
- Columns used in ORDER BY (`orders.created_at`, `customers.name`)

Good indexes can reduce the ids query from hundreds of milliseconds to single digits. See [Benchmark Results](/performance/benchmarks) for measured before/after data.
