Performance tuning has a correct order of operations. Doing things in the wrong order wastes time and can make things worse (as the benchmark data shows — adding pooling before fixing facets increased timeouts). Follow this sequence.

## Step 1 — Add indexes first

Before writing any custom strategies, ensure your database has indexes on the columns the engine touches most:

**Foreign key columns** — every column used in a JOIN condition:

```sql
CREATE INDEX ON order_lines(order_id);
CREATE INDEX ON order_lines(product_id);
CREATE INDEX ON orders(customer_id);
```

**Sort columns** — columns commonly used in ORDER BY:

```sql
CREATE INDEX ON orders(created_at);
CREATE INDEX ON customers(name);
```

**Filter columns** — columns frequently used in WHERE conditions:

```sql
CREATE INDEX ON orders(status);
CREATE INDEX ON orders(deleted_at) WHERE deleted_at IS NULL;
```

Based on the benchmarks, proper indexes on a relation-heavy resource can improve baseline throughput by ~8× and cut p99 latency by ~85%. This is almost always more impactful than any code change.

## Step 2 — Configure your connection pool explicitly

An implicit or default connection pool can flatten performance at medium concurrency (10–15 connections behaving as 3–4 effective). Set explicit pool limits:

```ts twoslash
// @noErrors
import { Pool } from "./twoslash-support";

// Example with pg driver
const pool = new Pool({
  max: 10, // cap concurrent connections
  min: 2,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
  statement_timeout: 10000, // kill queries over 10s
  query_timeout: 10000,
});
```

Add statement and query timeouts. Without them, a slow facet query during a traffic spike will hold a connection indefinitely and starve other requests.

::warning
Set `max` conservatively. The benchmarks showed that widening the pool from implicit ~10 to explicit 15 made heavy-facet scenarios significantly worse — more concurrent expensive queries hit the database simultaneously. Start at `max: 8–10` and raise it only if you see connection queueing rather than query slowness.
::

## Step 3 — Benchmark your actual workload

Do not guess which stage is slow. Measure it.

A simple approach: add logging or tracing around the ids query, rows query, and facet queries separately. Look for:

- ids query p99 > 200ms under moderate concurrency → candidate for `strategy.ids`
- facet queries > 1s per facet → review facet field types, consider `strategy.facets`
- rows query unexpectedly slow → check indexes on primary key and foreign keys

The automatic pipeline instruments each stage. You can also measure stages manually:

```ts twoslash
// @noErrors
const strategy = {
  ids: async ({ request, context, utils }: any) => {
    const start = performance.now();
    const result = await utils.executeIdsQuery({ request, context });
    console.log(`ids stage: ${performance.now() - start}ms`);
    return result;
  },
};
```

## Step 4 — Add `strategy.ids` when the ids query is the bottleneck

The ids stage is where most relation-heavy resources accumulate cost. The automatic CTE path is general-purpose — it handles any join shape but pays overhead for it. A hand-written query for your specific resource's join shape is almost always faster.

```ts twoslash
// @noErrors
import { and, count, customers, db, eq, orders } from "./twoslash-support";

const strategy = {
  ids: async ({ request, context, utils }: any) => {
    const where = utils.buildWhereClause({
      filters: request.filters,
      search: request.search,
    });
    const orderBy = utils.compileOrderBy(request.sorting);

    return {
      ids: [],
      rowCount: 0,
    };
  },
};
```

**The benchmarks show ~12× baseline improvement** from automatic to hand-tuned ids. For resources with deep join chains, this is the highest-leverage optimization available.

## Field-type costs {#field-type-costs}

Not all field paths cost the same. This table guides which fields to expose in defaults and allowed lists:

| Field type      | Example                        | Cost in ids query         | Cost in facets |
| --------------- | ------------------------------ | ------------------------- | -------------- |
| Root column     | `orders.status`                | Low                       | Low            |
| Direct relation | `customers.name` (many-to-one) | Low (1 JOIN)              | Moderate       |
| One-to-many     | `orderLines.quantity`          | Moderate (JOIN + GROUP)   | Moderate       |
| Many-to-many    | `tags.name` (via junction)     | High (2 JOINs + distinct) | High           |

Rules of thumb:

- **Root columns**: safe to expose in search defaults, sort, facets
- **Direct (many-to-one) relations**: safe for most uses, benchmark at scale
- **One-to-many paths**: use in search/filter carefully, add to `sort.disabled`, benchmark before adding to facets
- **Many-to-many paths**: add to `sort.disabled` by default, add to facets only if you have a `strategy.facets` that handles the aggregation cost

## What not to optimize prematurely

- Do not write a `strategy.ids` before benchmarking
- Do not raise pool `max` before fixing slow queries (more connections hitting a slow query = more timeouts)
- Do not remove facets from `allowed` without first trying a targeted index on the facet field
- Do not write `strategy.query` unless the staged pipeline genuinely does not fit your endpoint shape

## When facets are fundamentally too expensive

If a many-to-many facet is essential to your UI and benchmarks show it is consistently over 3–5 seconds at moderate concurrency, custom SQL strategies will help at the margins but cannot eliminate the fundamental cost of grouped aggregation through junction tables. At that point, consider:

- A separate pre-aggregated table or materialized view for that specific facet
- Moving to a search-oriented backend (Meilisearch, Elasticsearch, Typesense) for the faceted explorer specifically, while keeping `drizzle-resource` for the simpler table endpoints
- Throttling or rate-limiting facet requests on the API level
