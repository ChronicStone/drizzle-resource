# Tune Performance

Automatic mode is the recommended starting point, not a guarantee of optimal SQL for every resource.

## Start in automatic mode

The built-in pipeline gives you:

- typed field paths
- a consistent request contract
- built-in ids, rows, and facet execution
- fewer moving parts to maintain

This is usually a good fit when:

- the table is small or moderate in size
- queries stay mostly on root fields
- relation paths are single-valued
- facets are absent or light

## Benchmark realistic traffic before overriding anything

Measure at least:

- non-faceted requests
- cheap facet requests
- many-to-many facet requests
- deep-page requests
- concurrency, not just single-request latency

## Promote `strategy.ids` when ids and count dominate latency

For relation-heavy resources, ids selection is often the first stage worth optimizing.

Selected benchmark results from this repo's `employees` resource:

| Scenario | Auto RPS | Manual RPS | Auto p99 | Manual p99 |
|---|---:|---:|---:|---:|
| baseline, 10 concurrency | 17.2 | 219.7 | 928ms | 92ms |
| baseline, 25 concurrency | 15.1 | 229.6 | 2136ms | 183ms |
| baseline, 50 concurrency | 14.1 | 240.9 | 4504ms | 409ms |
| m2m skill facet, 25 concurrency | 4.7 | 8.2 | 6427ms | 3535ms |

These numbers are specific to the benchmarked resource, but they show the shape of the tradeoff:

- automatic mode is the right default
- relation-heavy resources still need measurement
- a tailored `strategy.ids` can materially improve latency without changing the public API

## Treat facets as a separate performance concern

Facet cost grows quickly when you combine:

- many-to-many paths
- broad search
- multiple requested facets
- deep offset pagination

General rules of thumb:

- root-column facets are usually cheapest
- single-valued relation facets are usually manageable
- many-to-many facets are usually the most expensive

## Roll out changes in the safest order

Follow this path for new resources:

1. Start in automatic mode.
2. Add indexes for filter, search, and sort hot paths.
3. Benchmark realistic endpoint traffic.
4. Add `strategy.ids` if ids and count are the bottleneck.
5. Evaluate `strategy.facets` only if facets remain the hotspot.

For measured examples from this repo, continue to [Benchmark Results](/guide/benchmark-results).
