# Customize Strategies

Start with the built-in automatic mode, then override only the stage that has a concrete requirement.

## Override ids before replacing the whole pipeline

`strategy.ids` is usually the highest-leverage customization point.

Use it when:

- ids and count are slower than row hydration
- relation-path sorting or filtering forces expensive joins
- you want custom pagination SQL while keeping default rows hydration

```ts [resource.ts]
const resource = engine.defineResource('employees', {
  relations: {
    department: {
      with: {
        company: true,
      },
    },
  },
  strategy: {
    ids: async ({ request, utils }) => {
      const whereClause = utils.buildWhereClause(request)
      const orderBy = utils.compileOrderBy(request.sorting)

      return {
        ids: ['emp_1', 'emp_2'],
        rowCount: 2,
      }
    },
  },
})
```

## Override rows when hydration shape must change

Use `strategy.rows` when you need:

- a projection instead of relational `findMany(...)`
- denormalized rows
- batched service calls or a non-Drizzle data source

## Override facets when grouped SQL becomes the hotspot

Use `strategy.facets` when:

- per-facet grouped SQL is too expensive
- you need precomputed buckets
- you need custom aggregation semantics

## Override the full query only when the staged model no longer fits

Use `strategy.query` when one backend call should own ids, rows, count, and optional facets together.

If `strategy.query` already returns `facets`, the package does not call `strategy.facets` again.

## Reuse the built-in utilities inside your strategy

Prefer the helpers exposed through `utils`:

- `utils.buildWhereClause(request)`
- `utils.compileOrderBy(request.sorting)`
- `utils.executeRowsQuery({ ids, request })`
- `utils.resolveFacets({ request, facets })`

This keeps your custom behavior aligned with the same request contract your callers already know.
