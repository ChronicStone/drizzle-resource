---
title: Strategies
description: When and how to override individual pipeline stages — ids, rows, facets, or the full query — using the built-in SQL helpers.
---

The engine runs a staged pipeline by default. Strategies let you replace individual stages with custom implementations — without changing the request/response contract or throwing away the rest of the engine.

## The four stages

```
request → [ids stage] → [rows stage] → [facets stage] → response
```

| Stage    | Default behavior                                 | Why you'd override                                   |
| -------- | ------------------------------------------------ | ---------------------------------------------------- |
| `ids`    | SQL CTE to select page ids + count               | Custom join shape, keyset pagination, hand-tuned SQL |
| `rows`   | `db.query.root.findMany` with declared relations | Non-relational row source, custom hydration shape    |
| `facets` | Grouped aggregate queries per facet              | Pre-aggregated data, custom bucket logic             |
| `query`  | Full staged pipeline                             | When the staged model no longer fits at all          |

Override `ids` first. It is the stage where most performance gains come from.

## Overriding `strategy.ids`

This is the most impactful override. The ids query selects the page of ids and the total count. A hand-tuned SQL query here can be 10× faster than the automatic path for relation-heavy resources.

```ts twoslash
// @noErrors
import { and, customers, count, db, eq, orders } from "./twoslash-support";

const strategy = {
  ids: async ({ request, context, utils }: any) => {
    const { pagination, sorting, filters, search } = request;

    const where = utils.buildWhereClause({ filters, search });

    return {
      ids: [],
      rowCount: 0,
    };
  },
};
```

Use `utils.buildWhereClause` to compile the normalized filters and search into a Drizzle `where` condition. The full utils API is listed in [Reference → Strategy Utils](/reference/strategy-utils).

## Overriding `strategy.rows`

Use this when you need a different hydration source or shape — a custom SELECT, a view, or a non-relational lookup:

```ts twoslash
// @noErrors
const strategy = {
  rows: async ({ ids, utils }: any) => utils.executeRowsQuery({ ids }),
};
```

The `ids` array is pre-ordered. Your rows query must return rows in the same order, or use `utils.executeRowsQuery` which handles deduplication and order restoration for you.

## Overriding `strategy.facets`

```ts twoslash
// @noErrors
const strategy = {
  facets: async ({ request, facets, context, utils }: any) =>
    utils.resolveFacets({ request, facets }),
};
```

## Overriding `strategy.query`

The nuclear option — replaces the entire pipeline. Only use this when the staged model genuinely does not fit your endpoint:

```ts twoslash
// @noErrors
const strategy = {
  query: async ({ request, context, utils }: any) => ({ rows: [], rowCount: 0 }),
};
```

## The built-in utils helpers

Every strategy override receives a `utils` object with compiled helpers that reuse the engine's field registry, filter compiler, and Drizzle bindings:

| Helper                                        | Purpose                                                     |
| --------------------------------------------- | ----------------------------------------------------------- |
| `utils.buildWhereClause({ filters, search })` | Compile a Drizzle `where` condition from normalized request |
| `utils.compileOrderBy(sorting)`               | Compile a Drizzle `orderBy` array from sorted entries       |
| `utils.executeIdsQuery(options)`              | Run the default ids + count query                           |
| `utils.executeRowsQuery({ ids })`             | Hydrate rows by ids using the declared relations            |
| `utils.resolveFacets({ request, facets })`    | Run default facet aggregation                               |
| `utils.resolveField(path)`                    | Resolve a field path to its Drizzle column reference        |

Using these helpers keeps your strategy consistent with the rest of the engine — you get typed field resolution, proper filter compilation, and relation-aware SQL without rebuilding it from scratch.

## When to override

Start without any strategy overrides. The automatic pipeline is correct and often fast enough for tables under 100k rows with good indexes. Add overrides only when you have benchmark data showing a bottleneck.

See [Performance → Optimization guide](/performance/optimization) for the decision flowchart.
