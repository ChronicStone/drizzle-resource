# Defining a Resource

`engine.defineResource(root, options)` creates a resource bound to one root table. It is the primary configuration surface of the package.

```ts
const ordersResource = engine.defineResource('orders', {
  relations: { ... },
  query: { ... },
  strategy: { ... },   // optional
})
```

## `root`

The table name as it appears in your schema map — for example `'orders'`, `'products'`, `'customers'`. This is the table the resource is built around.

## `relations`

The Drizzle `with` tree that controls which related tables are joined during row hydration, and which nested field paths become valid.

```ts
relations: {
  customer: true,
  orderLines: {
    with: {
      product: {
        with: { category: true },
      },
    },
  },
}
```

This does two things:

1. **Defines the row shape** — when the engine hydrates rows using `db.query.orders.findMany(...)`, it passes this tree as the `with` option, producing fully-joined row objects.
2. **Registers nested field paths** — `customer.name`, `orderLines.product.name`, `orderLines.product.category.label` all become valid field paths for search, sort, filter, and facet configuration.

::note
If you need a different join tree for the ids query (for performance), you can specify it in `strategy.ids`. The `relations` tree only controls the row hydration query and field registration.
::

## `query`

The declarative behavior of the resource. Everything in `query` can be left out — the engine falls back to reasonable defaults.

### `query.scope`

A function that merges additional filters into every request. Use this for tenancy constraints, authorization rules, and soft-delete conditions.

```ts
scope: (f, ctx) => f.and([f.is("customer.orgId", ctx.orgId)]);
```

The filter builder is already created for the resource and typed from its `relations`.

See [Scope](/resource-setup/scope) for details and multi-tenant examples.

### `query.search`

Which fields are searchable and what gets searched by default:

```ts
search: {
  allowed: ['reference', 'customer.name', 'orderLines.product.name'],
  defaults: ['reference', 'customer.name'],
}
```

`allowed` is the full set of valid search fields. `defaults` is what gets searched when the client sends `search.fields: []`. See [Query Contract → Search](/query-contract/search).

### `query.sort`

Default sort and disabled fields:

```ts
sort: {
  defaults: [{ key: 'createdAt', dir: 'desc' }],
  disabled: ['orderLines.product.name'],
}
```

`defaults` applies when the client sends an empty `sorting` array. `disabled` silently drops those keys if the client sends them. See [Query Contract → Sorting](/query-contract/sorting).

### `query.filters`

Fields the client cannot use in filter conditions:

```ts
filters: {
  hidden: [],    // field paths removed from the filter registry entirely
  disabled: [],  // field paths the client cannot filter on
}
```

`hidden` removes fields from the registry so they cannot be used in any client-supplied operation. `disabled` prevents them from being used as filter keys specifically.

### `query.facets`

Which fields can be requested as facets:

```ts
facets: {
  allowed: ['status', 'customer.name', 'orderLines.product.category.label'],
}
```

A facet request with a key not in `allowed` is rejected. See [Query Contract → Facets](/query-contract/facets).

### `query.defaults`

Fallback pagination when the client sends no pagination or `pageSize: 0`:

```ts
defaults: {
  pagination: { pageSize: 25 },
}
```

## `strategy`

Optional imperative overrides for any stage of the pipeline. See [Strategies](/resource-setup/strategies).

```ts
strategy: {
  ids: async ({ request, context, utils }) => { ... },
  rows: async ({ request, context, ids, utils }) => { ... },
  facets: async ({ request, context, facets, utils }) => { ... },
  query: async ({ request, context, utils }) => { ... },
}
```

Only override a stage when you have a measured performance reason to do so. The automatic pipeline handles the vast majority of cases correctly.
