# Field Policy

Field policy is the set of rules that govern what the client can do with each field path on a resource. It is configured declaratively in `defineResource` and enforced by the engine before any SQL runs.

## The four axes

Each field path on a resource can be controlled along four axes:

| Axis   | Config key                           | Effect                                         |
| ------ | ------------------------------------ | ---------------------------------------------- |
| Search | `search.allowed`, `search.defaults`  | Which fields appear in ILIKE search            |
| Sort   | `sort.disabled`                      | Which fields cannot be used as sort keys       |
| Filter | `filters.hidden`, `filters.disabled` | Which fields cannot be filtered on             |
| Facet  | `facets.allowed`                     | Which fields can be requested as facet buckets |

## Search policy

```ts twoslash
// @noErrors
import type { ExampleSearchConfig } from "./twoslash-support";

const query = {
  search: {
    allowed: ["reference", "customer.name", "orderLines.product.name"],
    defaults: ["reference", "customer.name"],
  },
} satisfies { search: ExampleSearchConfig };
```

If `allowed` is omitted, all registered field paths are considered searchable. This is rarely desirable in production — set `allowed` explicitly so clients cannot search on sensitive or expensive fields.

`defaults` should be the most useful search fields. Keep it narrow. Every additional default field adds an ILIKE condition to every search request.

## Sort policy

```ts twoslash
// @noErrors
import type { ExampleSortConfig } from "./twoslash-support";

const query = {
  sort: {
    defaults: [{ key: "createdAt", dir: "desc" }],
    disabled: ["orderLines.product.name"],
  },
} satisfies { sort: ExampleSortConfig };
```

Add many-to-many relation paths to `disabled`. Sorting through a junction table requires a join that can produce incorrect ordering without aggregation, and the cost is significant. Until you have a `strategy.ids` that handles it correctly, disable it.

```ts twoslash
// @noErrors
import type { ExampleSortConfig } from "./twoslash-support";

const query = {
  sort: {
    disabled: ["tags.name", "orderLines.product.category"],
  },
} satisfies { sort: ExampleSortConfig };
```

When a disabled sort key is submitted, it is silently dropped — not an error. The remaining sort entries still apply.

## Filter policy

```ts twoslash
// @noErrors
import type { ExampleFiltersConfig } from "./twoslash-support";

const query = {
  filters: {
    hidden: ["internalCostPrice", "marginPercent", "internalNotes"],
    disabled: ["deletedAt"],
  },
} satisfies { filters: ExampleFiltersConfig };
```

`hidden` is the stronger option — the field does not appear in the registered field set at all. Use it for sensitive columns that clients should not know exist.

`disabled` keeps the field registered (so scope functions and strategies can still reference it) but prevents it from appearing in client-supplied filter conditions.

::tip
Use `hidden` for sensitive financial, security, or internal-only columns. Use `disabled` when you want the engine to be able to use the field internally but prevent client filters from using it directly.
::

## Facet policy

```ts twoslash
// @noErrors
import type { ExampleFacetsConfig } from "./twoslash-support";

const query = {
  facets: {
    allowed: ["status", "customer.name", "orderLines.product.category.label"],
  },
} satisfies { facets: ExampleFacetsConfig };
```

Facets are opt-in. A client requesting a facet key not in `allowed` gets a validation error. This is important because facet queries add cost proportional to the number of facets and the complexity of the join path — you should only expose facets you have benchmarked.

## Combining policies

A field can have different policies on different axes. For example, `customer.orgId` might be:

- In `filters.hidden` (clients cannot filter on it directly — the scope handles it)
- Not in `search.allowed` (cannot be searched)
- Not in `facets.allowed` (cannot be faceted)
- Not in `sort.disabled` (irrelevant since it's hidden)

```ts twoslash
// @noErrors
const query = {
  filters: {
    hidden: ["customer.orgId"],
  },
  search: {
    allowed: ["reference", "customer.name"],
    defaults: ["reference", "customer.name"],
  },
  facets: {
    allowed: ["status", "customer.name"],
  },
};
```
