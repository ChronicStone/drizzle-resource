---
title: Relations
description: How the relations tree controls row hydration, field-path registration, and join behavior.
---

The `relations` option in `defineResource` controls two separate things: which related data gets loaded when hydrating rows, and which nested field paths become valid for search, sort, filter, and facet operations.

## Basic declaration

```ts twoslash
// @noErrors
import { engine } from "./engine";

engine.defineResource("orders", {
  relations: {
    customer: true, // join customer table
    orderLines: {
      with: {
        product: true, // join products through order_lines
      },
    },
  },
});
```

With this declaration:

- **Row hydration** — `db.query.orders.findMany({ with: { customer: true, orderLines: { with: { product: true } } } })` is the query that runs when hydrating a page of rows. Each row object will include `customer` and `orderLines[].product`.
- **Field paths** — `customer.id`, `customer.name`, `customer.orgId`, `orderLines.quantity`, `orderLines.product.name`, `orderLines.product.sku` all become valid field paths in the resource.

## What "valid field path" means

Any field that exists in a table reachable through the declared relations can be used as:

- a sort key in `sorting`
- a filter key in `filters` or any nested `group.children`
- a search field in `search.allowed` and `search.defaults`
- a facet key in `facets.allowed`

Paths not in the declared relations are rejected at validation time. This keeps the field registry explicit.

## Nested relations

You can go as deep as your schema allows:

```ts twoslash
// @noErrors
const relations = {
  customer: true,
  orderLines: {
    with: {
      product: true,
    },
  },
};
```

This registers paths like `customer.org.country.name` and `orderLines.product.category.label`.

## Relation types and their implications

**One-to-one / many-to-one** (e.g. `orders.customer`) — a single JOIN. Cheap for sort, filter, and facets. The row structure is a flat object on each row.

**One-to-many** (e.g. `orders.orderLines`) — each order row gets an array of order lines. Filtering and sorting through one-to-many paths requires a subquery or join with potential row duplication. The engine handles this, but the cost grows with the number of children per row.

**Many-to-many** (through a junction table, e.g. `products.tags` through `product_tags`) — most expensive. Requires a join through the junction table. Sorting is ambiguous, facet aggregation uses `count(distinct)`, and search generates a subquery. These should be benchmarked explicitly before exposing to clients.

## Using a different join tree for performance

If you want a lighter join tree for the ids query (to avoid expensive joins during page-id selection), declare the full tree in `relations` for row hydration, and override the ids stage with a simpler join in `strategy.ids`:

```ts twoslash
// @noErrors
import { engine } from "./engine";
import { request, utils } from "./twoslash-support";

engine.defineResource("orders", {
  relations: {
    customer: true,
    orderLines: { with: { product: true } },
  },
  query: {},
  strategy: {
    ids: async ({ request, utils }) => {
      return utils.executeIdsQuery({ request });
    },
  },
});
```

This is the primary performance optimization for relation-heavy resources. See [Performance → Strategies](/performance/optimization).
