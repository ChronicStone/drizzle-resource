# Scope & Security

The scope is a function that runs before every query and injects additional filters the client cannot see, override, or bypass. It is the primary mechanism for tenancy, authorization, and record-visibility constraints.

## Why scope exists

Without scope, you would need to remember to add a tenant filter in every controller that calls `resource.query`. Miss one, and a tenant can read another tenant's data. Scope ensures the constraint is structural — it cannot be forgotten, it cannot be bypassed by a clever filter payload.

## Defining a scope

Use the resource's inline `query.scope(filters, context)` callback:

```ts
query: {
  scope: (f, ctx) =>
    f.and([
      f.is('customer.orgId', ctx.orgId),
    ]),
}
```

The filters produced by `scope` are AND-merged into every request's top-level filter array before SQL runs. Client filters sit alongside them — they cannot reach outside the scope.

## Scoping through relations

If your tenancy key is on a related table, include that join path in the resource `relations` config so the builder exposes it:

```ts
relations: {
  customer: true,
},
query: {
  scope: (f, ctx) => f.and([
    f.is('customer.orgId', ctx.orgId),
  ]),
}
```

## Multi-condition scope

Scope can enforce multiple conditions — tenancy plus soft-delete, for example:

```ts
scope: (f, ctx) => f.and([f.is("customer.orgId", ctx.orgId), f.is("deletedAt", null)]);
```

## Returning no constraint

If a request context allows unrestricted access (e.g. an internal admin user), the scope function can return an empty group:

```ts
scope: (f, ctx) => {
  if (ctx.isAdmin) return f.and([]); // no restriction
  return f.and([f.is("customer.orgId", ctx.orgId)]);
};
```

## Reusing a scope

If you want to share the same scope across resources, `engine.defineScope(...)` still works as a compatibility helper and returns an inline-compatible `scope(filters, context)` callback.

## What the client cannot do

The scope merges at the engine level, before the request's filter tree is processed. The client-supplied `filters` tree is always AND-ed with the scope output. There is no way for the client to:

- Remove a scope condition with an OR at the top level
- Bypass the scope by sending a conflicting filter
- See that a scope exists or what fields it constrains

## Scope vs hidden fields

Scope is for enforcement — it guarantees records are filtered even if no client filter is applied.

`filters.hidden` and `filters.disabled` are for access control at the field level — they prevent clients from filtering or querying on fields they should not know about. Use both where appropriate:

- **Scope** — restrict which rows are accessible
- **`hidden`** — remove sensitive fields from the query registry entirely (e.g. `internalNotes`, `costPrice`)
