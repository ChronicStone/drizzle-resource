# Overview

This section is the formal API reference.

If you want conceptual guidance first, start with:

- [Query Contract](/query-contract/overview)
- [Configuration](/configuration/overview)

Come here when you want the runtime surface and type-level contract spelled out more directly.

## Start from the package entry point

```ts [index.ts]
import {
  createQueryEngine,
  createQueryFilterBuilder,
} from '@exassess/drizzle-query-resource'
```

The package exports:

- runtime factories such as `createQueryEngine` and `createQueryFilterBuilder`
- resource methods such as `query`, `queryIds`, `queryRows`, and `queryFacets`
- public types for requests, responses, field paths, strategies, and resources

## Understand what `createQueryEngine(...)` really creates

```ts [engine.ts]
import { createQueryEngine } from '@exassess/drizzle-query-resource'
import db from '#db'
import * as schema from '#generated/db/schemas'
import { relations } from '#generated/db/relations'

const engine = createQueryEngine({
  db,
  schema,
  relations,
}).withContext<{ orgId: string }>()
```

`createQueryEngine({ db, schema, relations })` binds the package to one Drizzle world:

- `db` gives it query execution
- `schema` gives it table and column metadata
- `relations` gives it relation traversal and nested field typing

From that engine, you get a factory object that can:

- narrow shared context with `.withContext<T>()`
- create typed scope builders with `defineScope(...)`
- define resources with `defineResource(...)`

## Understand what a resource contains at runtime

A resource is more than a function you call.

It contains:

- `key` for the root table
- `relations` for default hydration
- `fields` for the resolved field registry
- `queryConfig` for normalized runtime config
- the public methods `query`, `queryIds`, `queryRows`, and `queryFacets`

That is why the engine can normalize and validate requests consistently before execution.

## Understand the execution model behind the methods

The built-in query flow is staged:

1. normalize request defaults
2. merge scope filters
3. validate fields
4. select ids and total count
5. hydrate rows
6. optionally resolve facets

That staged model is reflected in the public methods:

- `query(...)` runs the full page flow
- `queryIds(...)` stops after ids and total count
- `queryRows(...)` hydrates a known ordered id list
- `queryFacets(...)` resolves facets independently

## Use `createQueryFilterBuilder(...)` when you need typed filter objects

The filter builder is the low-level helper for constructing filter nodes without dropping field-path typing.

That is most useful in:

- `query.scope`
- shared helper functions
- tests or request-building utilities
